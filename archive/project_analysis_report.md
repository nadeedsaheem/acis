# ACIS GraphRAG Project: Comprehensive Technical Report

## 1. Executive Summary

The ACIS GraphRAG project is a sophisticated Retrieval-Augmented Generation (RAG) system built to provide intelligent, natural language querying over the ACIS C++ codebase. The system employs a multi-stage pipeline that parses raw C++ header files into a structured dataset, constructs a Neo4j Knowledge Graph, generates vector embeddings for semantic search, and exposes a production-ready FastAPI service backed by Large Language Models (LLMs). 

The goal of the system is to allow developers to ask complex questions about the ACIS codebase (e.g., "How does variable radius blending work?" or "Which classes inherit from ENTITY?") and receive accurate, grounded answers strictly derived from the codebase's documentation and topological structure.

---

## 2. Architecture Overview

The system architecture is divided into four primary stages:

1. **Extraction (Parsing):** A highly resilient Tree-Sitter parser reads C++ headers and handles complex macro obfuscations to extract classes, structs, enums, methods, functions, and inheritance relationships into a structured JSON dataset.
2. **Knowledge Graph Construction:** The parsed JSON is ingested into a Neo4j graph database, creating a rich network of interconnected entities (Nodes) and their relationships (Edges).
3. **Vectorization & Semantic Search:** Documentation and entity payloads are embedded using SentenceTransformers (`BAAI/bge-small-en-v1.5`). A Neo4j Vector Index is built, and a hybrid search strategy (Vector + Lexical) is implemented for accurate retrieval.
4. **RAG & API Layer:** A FastAPI web service orchestrates user queries. Queries are normalized, relevant context is retrieved from Neo4j, structured into Markdown, and sent to an LLM with strict grounding instructions to generate the final answer.

## 4. Code Documentation: Component Analysis

The source code in the `src/` directory is modular and divided by domain responsibilities.

### 4.1 Data Extraction
* **`multi_repo.py`**
  * **Role:** The production C++ AST parser.
  * **Implementation:** Uses Tree-Sitter (`tree_sitter_cpp`) to traverse C++ files. It features robust heuristic fallbacks to bypass C++ macro obfuscations (like `DECL_KERN` or `MASTER_ATTRIB_DECL`) ensuring accurate entity resolution. It maps classes, inheritance, parameters, return types, and rigorously cleans documentation strings.
  * **Output:** Generates `code_base.json`.

### 4.2 Knowledge Graph Construction
* **`build_graph.py`**
  * **Role:** Ingests the JSON dataset into a Neo4j database.
  * **Implementation:** Defines a `KnowledgeGraphBuilder` that batches transactions for performance. It enforces schema constraints and handles the creation of nodes (`File`, `Class`, `Method`, `Function`, `Parameter`, `Documentation`, etc.) and complex edges (`CONTAINS`, `INHERITS`, `HAS_METHOD`, `HAS_PARAMETER`, `RETURNS`, `USES_TYPE`). Identity is managed using an SHA-256 hash of file paths and signatures to prevent collisions.

### 4.3 Vector Embedding & Retrieval
* **`embed_docs.py`**
  * **Role:** Embedding generation and semantic search execution.
  * **Implementation:** Loads `SentenceTransformer` (`BAAI/bge-small-en-v1.5`) to vectorise `Documentation` nodes. Creates a Neo4j Vector Index (`documentation_embedding_index`). Implements `semantic_search()` using a hybrid approach—combining vector cosine similarity via Neo4j's `SEARCH` with lexical matching to prevent generic keywords from flooding results.
* **`rebuild_embeddings.py`**
  * **Role:** Maintenance utility to rebuild embeddings by concatenating entity context (Entity Type, Class, Signature) with documentation, improving vector search accuracy.
* **`retriever.py`**
  * **Role:** Implements exact-match and lexical retrieval strategies.
  * **Implementation:** Uses Cypher queries to perform fast exact-match lookups on entity names and signatures. Expands results by fetching surrounding context (parameters, parent classes, enum values). Provides a lazy-loading wrapper for semantic search.

### 4.4 GraphRAG Pipeline
* **`query_normalizer.py`**
  * **Role:** Query pre-processing.
  * **Implementation:** Detects simple entity lookup queries (e.g., "SPAposition") and normalizes them into questions ("What is SPAposition?") after verifying the entity exists in Neo4j.
* **`context_builder.py`**
  * **Role:** Context management.
  * **Implementation:** Sorts retrieved Neo4j results by score and priority, converting the structured JSON/dict data into concise Markdown blocks. Enforces a `max_chars` limit to prevent LLM context overflow.
* **`llm_provider.py`**
  * **Role:** LLM Abstraction Layer.
  * **Implementation:** Detects available API keys (OpenAI, Gemini, Anthropic) and routes prompts accordingly. Includes a mock/fallback mode for testing validation queries when APIs are unavailable or rate-limited.
* **`graphrag_service.py`**
  * **Role:** The core RAG orchestrator.
  * **Implementation:** Defines `GraphRAGService.answer_question(query)`. It times the retrieval, calls `semantic_search()`, builds context, constructs strict grounding prompts, and triggers the LLM to generate an answer. Collects metadata on used sources and processing time.

### 4.5 API Layer
* **`api_models.py`**
  * **Role:** Defines Pydantic models (`QueryRequest`, `QueryResponse`, `HealthResponse`, `ErrorResponse`) for strict API validation and schema definition.
* **`api_server.py`**
  * **Role:** FastAPI application server.
  * **Implementation:** Exposes a `/health` endpoint that verifies Neo4j connectivity, and a `/query` endpoint that handles POST requests. It wraps the RAG service execution in an `asyncio.wait_for` timeout, gracefully handles exceptions, and tracks latency metrics.

## 5. Knowledge Graph Schema

The Neo4j database utilizes the following node labels and relationships:

### Nodes:
* **`File`**: Represents a physical source file (`path`).
* **`Class` / `ExternalClass`**: C++ Classes.
* **`Struct`**: C++ Structs.
* **`Enum` / `EnumValue`**: Enumerations and their specific values.
* **`Function` / `Method`**: Executable blocks of code, categorized by ownership.
* **`Parameter`**: Input arguments to functions/methods.
* **`PrimitiveType` / `ExternalType`**: Return types and basic types.
* **`Documentation`**: Contains the cleaned docstrings and vector `embedding` array.

### Edges (Relationships):
* `(File)-[:CONTAINS]->(Class | Function)`
* `(Class)-[:INHERITS]->(Class | ExternalClass)`
* `(Class)-[:HAS_METHOD]->(Method)`
* `(Function | Method)-[:HAS_PARAMETER]->(Parameter)`
* `(Function | Method)-[:RETURNS]->(Class | PrimitiveType | ExternalType)`
* `(Parameter)-[:USES_TYPE]->(Class)`
* `(Entity)-[:HAS_DOC]->(Documentation)`
* `(Enum)-[:HAS_VALUE]->(EnumValue)`
